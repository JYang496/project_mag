project_mag 后续玩法优化头脑风暴报告

生成日期：2026-07-02 | 范围：基于最近一次 git pull 后的未提交改动、现有 roguelike 优化阶段文档、构筑验证矩阵和当前 UI/奖励/HUD 改造面。

这份报告不是新的实现计划，而是把当前改动暴露出来的机会点整理成下一轮可讨论、可拆分、可验证的优化方向。核心判断：项目已经从“功能能跑”进入“构筑选择要能被玩家理解、被关卡挑战验证、被报告数据追踪”的阶段。

当前基础

最近改动已经建立了几条后续优化可以依赖的基础设施：标准战斗奖励开始允许模块进入 draft，奖励卡和仓库/商店 UI 能显示构筑短标签与适配状态，HUD 资源表达从文字转向图形化控件，路线选择开始展示风险/收益指标，构筑验证矩阵已经把 6 个构筑与 6 类遭遇题型写成报告。

因此，下一轮不应优先继续堆更多文案或单点 UI，而应把“玩家如何形成构筑、路线如何施压、敌人如何检验构筑、报告如何量化结果”连成闭环。

优先级总览

优先级 方向 为什么现在做 主要落点 验证方式
P0 构筑验证从声明矩阵升级到实测矩阵 当前 `build_validation_matrix.html` 是声明强弱，不是实测 DPS/生存/击杀节奏。 data/test/build_validation_matrix_default.tres、tests/headless/combat、docs/reports/ 每个构筑 x 遭遇输出击杀数、存活时间、伤害来源占比、资源异常。
P0 奖励 draft 节奏校准 模块奖励已经进入普通战斗奖励池，需要证明不会稀释武器进度，也不会让模块过早成型。 World/rewards/reward_manager.gd、data/EconomyConfig.gd、docs/design/reward_draft_rules.md 模拟前 10 场奖励，记录武器进度、模块数量、经济项出现率和重复过滤。
P1 路线真正改变遭遇主题 路线卡 UI 已经能展示指标，但路线如果只改 HP/奖励倍数，玩家决策仍然薄。 data/routes/*.tres、data/spawns/spawn_combat_profile.tres、World/spawn/ Normal/Difficult/Bonus 在敌人组合、压力曲线、奖励倾向上有可观测差异。
P1 主动技能能量与武器资源形成明确取舍 HUD 现在突出主动技能能量成本，但玩法层面还需要说明何时值得留能量、何时值得花能量。 Player/Skills/skills.gd、player_active_skill_runtime.gd、HUD resource meter tests 技能释放频率、能量短缺窗口、武器 energy 消耗路径均有测试和 UI 状态。
P2 战斗中构筑反馈 奖励/管理 UI 已经解释构筑，但战斗中玩家还需要知道“我的构筑正在生效”。 UI/scripts/components/hud_presenter.gd、被动触发事件、状态效果提示 每个构筑轴至少有一个低噪音的战斗反馈点，如 Heat spend、Mark execute、Freeze shatter。

方向一：构筑实测验证矩阵

目标

把当前声明式矩阵升级为可运行的 smoke/benchmark 矩阵。每个构筑不只写“Strong/Weak”，而是在固定遭遇中输出实际表现：清怪速度、精英击杀时间、受击次数、资源溢出/短缺、核心武器贡献。

Heat LoopMark ExecuteFreeze ControlClose RiskArea ControlReload Rhythm

可交付内容

  • 新增 measured report：docs/reports/build_validation_measured_*.html。
  • 为 6 个构筑生成固定装配，不依赖玩家运行时随机奖励。
  • 为 6 个遭遇题型生成可重复压力场景。
  • 报告中保留 declared vs measured 对照，直接标出不符合预期的构筑/遭遇。

风险

  • 如果测试场景太接近靶场，会误判 Dash、近战、远程压力的真实风险。
  • 如果只测 DPS，会低估控制、位移、生存和任务目标压力。
  • 如果构筑装配绕过实际资源约束，会产生不可在真实 run 中复现的结果。

方向二：奖励 draft 节奏

模块进入标准战斗奖励池是正确方向，但它会立刻改变 run 的早期节奏。下一步需要回答：玩家在第 1-3 场是否仍能稳定获得武器进度？模块是否太早形成完整体系？经济奖励是否只是“没东西给时的填充”？

问题 建议 判定标准
模块奖励稀释武器升级 保留至少一个武器进度候选；模块与经济竞争剩余槽位。 前 5 场模拟中不会出现连续多场完全没有武器进度。
模块适配信息只在 UI 层可见 奖励生成报告同时记录模块标签、当前武器适配、被过滤原因。 报告能解释每个候选为什么出现或为什么被过滤。
任务奖励和普通奖励身份混淆 继续保持任务模块/地形效果不进入普通战斗奖励。 回归测试明确断言标准战斗奖励不含 task/cell effect。

方向三：路线与遭遇主题绑定

路线选择 UI 已经从文字按钮升级成信息卡片，这让“路线身份”有了承载面。现在缺的是让路线实际影响战斗题型，而不是只表现为倍率。

Normal

标准路线应保持综合压力，适合作为矩阵基线。奖励不偏科，敌人题型按默认关卡曲线推进。

baselinemixed pressure

Difficult

困难路线应显式提高某类题型压力，例如支援敌、远程压制、精英高血量，而不是单纯提升 HP。

supporteliteranged

Bonus

Bonus 应继续保持无战斗补给路线身份，可偏向经济修补、模块整理或构筑补齐，但不应比 Difficult 奖励更强。

no combatrepair build

验证

路线报告应输出每条路线实际触发的敌人组合、奖励倾向和历史恢复结果，而不是只检查资源字段存在。

route historyspawn pressure

方向四：战斗中的构筑反馈

当前奖励/仓库/商店 UI 已经开始让玩家在战斗外理解构筑。下一步应在战斗中用低噪音反馈告诉玩家“这条构筑轴确实在工作”。这不需要大 HUD 重做，优先做事件驱动的短状态。

构筑轴 战斗反馈建议 避免事项
Heat Shared heat spend、overheat lock、heat capacity gain 用 HUD meter 或短 buff 状态表达。 不要把每次热量变化都弹文字。
Mark / Execute 被标记目标死亡时显示一次 execute payoff，报告记录执行伤害贡献。 不要让普通命中也触发同级别反馈。
Freeze 冻结控制窗口和破碎爆发分开表现。 不要只用蓝色伤害数字替代机制反馈。
Reload Rhythm reload-triggered 模块触发后显示短暂 reload chain 状态。 不要把 reload 本身和 reload payoff 混在同一个提示里。

方向五：玩家辅助与操作教学闭环

操作提示已经从“面板整体点击”改成明确按钮和 Adaptive/Always/Hidden 模式。后续优化应把操作提示和当前系统压力绑定：玩家为什么此刻要切武器、为什么要整理模块、为什么要走困难路线。

建议新增“情境提示预算”：同一类提示在一次 run 中最多主动展开有限次数，之后只保留 compact summary。这样可以保留新手引导，同时避免熟练玩家被反复打断。

  • 战斗内：只提示和当前失败风险相关的操作，例如主武器正在 reload 且副武器可用时提示切换。
  • 奖励后：只解释新出现的奖励类型或第一次看到的不适配模块。
  • 准备阶段：仓库/任务/路线提示按上下文记忆折叠状态，不重复展开同一文本。

推荐执行顺序

先做“奖励节奏模拟报告”

这是最小、最高收益的下一步。它能直接验证模块进入普通奖励池后，武器进度、模块出现率、经济 fallback 是否合理。输出 HTML 报告，不先改数值。

再做“构筑实测 smoke 矩阵”

沿用现有 BuildValidationMatrix，先做低成本 smoke 指标，再逐步接近真实战斗。关键是让 declared 强弱和 measured 结果可以对照。

然后绑定路线与遭遇主题

在有奖励节奏和构筑矩阵之后，再改路线/敌人压力，能避免路线设计变成主观调参。

最后补战斗内反馈

当构筑轴和遭遇题型稳定后，再决定哪些事件值得显示，避免 HUD 提示跟随尚未稳定的玩法语义来回改。

建议新报告族

报告 用途 输入 输出判断
reward_draft_simulation_*.html 验证每场战斗后的奖励候选结构。 EconomyConfig、RewardManager、当前玩家武器/模块状态。 是否保底进度、是否过多重复、是否模块过早成型。
build_validation_measured_*.html 把构筑声明矩阵转为可观测结果。 BuildValidationMatrix、固定遭遇、固定构筑装配。 declared strong/weak 是否符合实测。
route_pressure_profile_*.html 证明路线差异真实影响敌人和奖励。 RunRouteDefinition、SpawnCombatProfile、RewardManager。 路线是否产生可解释的风险/收益身份。
combat_feedback_signal_audit.html 梳理哪些被动/模块/状态事件值得进入 HUD。 weapon passives、module hooks、HUD presenter、现有测试。 每个构筑轴是否有一个低噪音反馈点。

不建议现在做的事

  • 不要先大幅调武器数值。当前更缺的是奖励节奏和遭遇验证，不是单武器 DPS 表。
  • 不要把 UI 做成“最佳推荐系统”。当前已锁定的是适配状态和短标签，不是全局 build ranking。
  • 不要一次性新增大量敌人。先让现有敌人与 spawn 权重形成题型，再判断行为缺口。
  • 不要把 Bonus 路线改成另一种高收益战斗路线。它现在更适合作为构筑修补/经济补给身份。

下一步可直接拆分的 Worker 任务

Worker A：奖励模拟报告

只读/少量新增工具脚本，输出 docs/reports/reward_draft_simulation_*.html。重点统计前 10 场奖励候选，不改 gameplay 数值。

Worker B：构筑实测 smoke

扩展 data/test/build_validation_matrix_default.tres 的消费路径，建立最小可运行测量，不追求最终平衡。

Worker C：路线压力审计

审计当前路线字段到 spawn/reward 的真实调用链，列出路线差异哪些只是 UI、哪些真的影响运行。

Integrator：报告汇总

整合三个报告，决定先改奖励权重、遭遇主题还是 HUD 反馈。验证以 --check-only 和 focused scene-backed tests 为准。
