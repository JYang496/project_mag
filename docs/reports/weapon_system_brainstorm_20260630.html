武器系统头脑风暴回顾报告

生成日期：2026-06-30 ｜ 目的：沉淀当前武器开发思路，作为后续分支、模组、奖励与测试设计的回顾材料。

本报告总结当前武器系统的建议方向：保持武器本体、分支、模组、武器组合四层职责清晰，避免所有成长路径都退化成直接增伤；同时让 DPS 测试从总榜比较升级为按职责、场景和输出形态比较。

核心结论

推荐继续采用四层职责：武器本体 = 手感和触发源、分支 = 玩法路线、模组 = 中后期输出引擎、武器组合 = 后期上限。

最大的设计风险不是某一把武器数值偏高，而是分支、模组、融合都变成同一种“加伤害”。短期容易平衡，长期会让构筑只剩 DPS 排名。

保留差异

每把武器都要有可感知的输出方式、节奏、命中形态和协同钩子。不要让所有分支都只是在同一个攻击模板上换倍率。

测试分组

DPS 报告应按输出职责分组比较。贴脸爆发、持续光束、范围覆盖、蓄力窗口、状态 payoff 不应只放进一个总榜判断强弱。

职责分层

系统层 主要职责 推荐边界
武器本体 决定手感、射击节奏、命中形态、基础事件来源。 负责“这把武器为什么和别的武器不同”，不要承载太多后期复杂成长。
武器分支 改变玩法路线，例如单体、覆盖、控制、防御、状态入口或资源消耗。 优先改变条件和形态，少做纯倍率分支；同一武器的两个分支不要强化同一条轴。
模组 把武器事件、trait 和状态转换为中后期输出引擎。 强模组应依赖条件、trait 或状态消耗，而不是所有武器都能稳定吃满的常驻增伤。
武器组合 形成后期构筑上限，例如热量生成与消耗、冻结控制与爆发、标记与处决。 组合价值要能被玩家读懂，伤害来源、触发条件和 payoff 都需要明确归因。

DPS 测试解读

新的分支 DPS 测试已经解决“武器无法完整输出”的基础问题，但它仍然是特定窗口下的单目标测量。后续不建议只看总榜来改数值，而应按职责建立多个测试类别。

近战/贴脸单体 默认单体 多目标/覆盖 蓄力/窗口爆发 状态/资源 payoff 移动目标/追踪

判断原则：Shotgun、Dash Blade、Chainsaw 这类近距离武器在贴脸测试里高是合理信号；Orbit、Glacier、Prism Fan 这类覆盖型武器如果单体 DPS 不高，也不一定代表弱。报告需要展示它们各自负责的场景。

武器身份建议

武器/武器族 建议主身份 建议协同钩子
Machine Gun 高频触发器，稳定输出与热量/命中事件来源。 热量生成、on-hit 模组、护盾或支援分支。
Flamethrower / Glacier 范围持续压制，状态铺设，近中距离区域控制。 燃烧、冻结、减速、状态扩散和控制 payoff。
Plasma Lance / Cannon / Sniper 精准、穿透或窗口爆发，承担资源消耗后的 payoff。 热量消耗、标记、破甲、reload 节奏和处决。
Shotgun / Dash Blade / Chainsaw 近距离风险回报，强调位置、贴身窗口和高命中密度。 破甲、冰冻近战协同、防御补偿、击杀连锁。
Laser / Orbit / Charged Blaster 几何控制，依赖角度、锁定、光束持续或覆盖形态。 追踪、折射、分裂、持续照射、多目标覆盖。

分支设计建议

Machine Gun

一个分支走热量/触发引擎，一个分支走护盾/支援/压制。避免两个分支都只是提高射速或伤害。

Charged Blaster

Focus Lance 可以保留单体高爆发定位；Prism Fan 应该赢在覆盖、角度和多目标，而不是和 Focus Lance 比单体 DPS。

Shotgun

如果 Shatter Buckshot 和 base 单体 DPS 接近，应通过破甲、碎裂、冻结或近战协同拉开玩法差异。

Laser

Tracking Lens 这类改变命中方式并影响后续几何行为的分支值得保留。它比简单倍率更容易形成可读构筑。

模组预算

模组类型 预算建议 设计说明
通用常驻属性 低 适合作为基础补强，不应成为最优解。
特定 trait 限制 中 需要玩家围绕武器标签或分支路线选择。
跨武器条件协同 高 依赖两把或多把武器互相喂事件、状态或资源。
状态消耗 payoff 最高 需要先铺设状态，再消耗状态产生爆发，适合做后期构筑核心。

强模组优先读取真实事件和状态，例如 hit、damage_dealt、reload_started、heat、freeze、mark、kill。避免依赖武器 id 写死效果。

融合与奖励节奏

融合职责

融合更适合负责增加 trait、解锁分支、打开新兼容性或改变行为入口。武器等级上限仍应由武器数据本身决定。

奖励节奏

武器池和模组池保持分离。早期给主武器和基础模组，中期帮助重复武器与 trait 成型，后期放出跨武器协同和稀有 payoff。

稀有度建议影响出现时机、组合门槛和复杂度，而不是直接等同于强度。否则玩家会只看颜色，不看构筑机制。

技术与测试护栏

  • 分支逻辑继续走 WeaponBranchBehavior 这类扩展点，避免在武器脚本里硬编码分支 id。
  • 光束、散射、多段命中等几何逻辑要尽量复用公共路径，降低命中检测分散导致 DPS 为 0 的风险。
  • DPS 报告应记录 target_distance、分支 id、触发次数、命中次数和伤害来源归因。
  • 每次新增分支时，至少跑一个能完整输出的 headless 场景测试，再看数值是否需要调整。
  • 运行时避免用大量重复 get_nodes_in_group("enemies") 扫描作为武器主逻辑依赖。

推荐后续行动

建立武器身份矩阵

为每把武器写明主身份、输出形态、协同钩子、推荐测试类别和不应跨越的设计边界。

改造 DPS 报告视角

保留总榜，但新增按职责分组的排名和说明，避免用单体 DPS 误判覆盖型或控制型武器。

审计分支差异

检查每个分支是否真的改变玩法条件。对“表现几乎等同 base”的分支补上状态、几何、资源或协同差异。

沉淀三类协同模板

优先沉淀状态铺设/消耗、reload 支援/payoff、热量生成/消耗三类模板，后续武器和模组按模板扩展。

一句话回顾

武器系统的目标不是让每个分支在同一张 DPS 总榜上公平，而是让每条构筑路线在自己的战斗场景里有明确价值、可读触发和可验证输出。
