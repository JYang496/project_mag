标准战斗奖励 Draft 节奏详细设计

生成日期：2026-07-02 | 范围：方向二“奖励 draft 节奏”详细设计 | 状态：设计已锁定，尚未实施。

目标是让标准战斗奖励先保障早期武器骨架，再逐步引入模块构筑信号。标准战斗完成后的主要结算奖励改为 Draft，并替代当前自动生成的标准战斗地面 weapon/module drop，避免玩家同时获得掉落和额外三选一。任务奖励、特殊来源掉落、Bonus 无战斗路线和敌人配置不属于本轮节奏门控。

核心结论

前 3 次标准战斗奖励 draft 使用早期节奏门控：固定保留 1 个 weapon-progress 槽；第 1 次不出模块和正常经济；第 2 次模块按 20% 进入非保底槽；第 3 次进入正常模块概率；前 3 次禁用正常经济权重，但允许最终防空池经济兜底。第 4 次起完全回到长期权重，不加 pity。

标准战斗完成后不再自动额外生成一份地面 weapon/module drop。Draft 是标准战斗的主要奖励结算，不是叠加奖励。

Weapon-Progress 定义

weapon-progress 是一个统一候选池，而不是只代表已有武器升级。它包括：

  • 已有武器升级候选：目标武器未满级。
  • 已有武器融合候选：目标武器可 fuse。
  • 新武器获得候选：普通武器 reward candidate，排除隐藏、零权重、已最终 fuse 的武器。

早期保底槽在这个统一池内按各自 drop_weight 抽 1 个。已有武器升级/融合使用目标武器定义的 rarity/weight；新武器沿用 WeaponDefinition 的 rarity/weight。

这意味着前 3 次并不强制推高当前主武器。设计选择保留探索感：已有武器成长和新武器获得同池竞争。

标准掉落替换规则

当前 RewardManager 在 BATTLE -> PREPARE 时会调用 _spawn_completed_battle_drops()，然后通过 build_completed_battle_drop_rewards() 自动生成地面 weapon/module drop。实现本设计后，标准战斗完成路径应改为打开标准 Draft，并停止这条自动地面掉落结算，避免同一场标准战斗既掉装备又弹三选一。

这不是删除所有掉落系统，而是把“标准战斗完成奖励”的结算形式从地面掉落改为 Draft。

配置建议

扩展现有 EconomyConfig，因为当前奖励概率、模块开关和经济奖励数值已经在那里。不要新增独立 config resource，除非后续路线差异化 draft 大幅复杂化。

字段建议 默认值 用途
early_standard_draft_count 3 早期门控覆盖前几次标准战斗 draft。
early_weapon_progress_slot_enabled true 早期是否固定保留 1 个 weapon-progress 槽。
early_module_option_chances [0.0, 0.2, -1.0] 每个早期阶段非保底槽的模块概率；-1.0 表示使用长期概率。
early_economy_option_enabled false 早期是否允许正常经济权重。
early_allow_fallback_economy true 武器/模块候选无法填满时是否允许经济兜底。

RewardManager 接口方向

RewardManager 不应持有长期状态。它应在生成标准战斗 reward options 时向 RewardDraftRuntime 请求当前阶段上下文，例如：

  • get_standard_draft_index()：当前将要生成的是第几次标准 draft。
  • get_stage_rules(economy)：返回模块概率、经济是否可用、是否需要 weapon-progress 槽。
  • set_pending_standard_draft(options, context)：在交给 UI 前持久化选项。
  • record_standard_draft_consumed()：UI 打开成功或 fallback grant 后递增。
  • clear_pending_standard_draft()：选择奖励成功后清除。

不要把阶段判断散落在 _roll_reward_option() 多个分支里。推荐先构建一个 stage rules dictionary，再由现有候选池和 roll 函数消费。

UI 规则

不在奖励面板上额外说明第 1/2/3 次后台节奏，也不在模块第一次出现时加专门教学提示。玩家只通过奖励结构本身感知节奏。

UI 项 规则
阶段标签 不显示 不显示 Early Draft / Weapon Growth 等后台规则。
模块首次出现提示 不新增 沿用奖励卡已有 fit/tag/detail 信息。
奖励卡内容 继续使用现有类型、稀有度、名称、短标签、落点说明。

奖励模拟报告验收

第一版实施必须包含固定 RNG seed 的模拟报告，报告要有宽松统计阈值，用来抓明显偏离，而不是把正常随机波动判失败。

指标 目标 建议阈值
样本 固定 seed，模拟 1000 次 run 的前 6 次标准 draft。 报告显示 seed、样本数、配置快照。
前 3 次 weapon-progress 每次都有 1 个 weapon-progress 槽。 必须 100%。
第 1 次模块出现率 0%。 必须 0%。
第 2 次模块出现率 约 20%。 接受 15%-25%。
第 3 次模块出现率 接近长期模块率。 目标值约 ±7%。
前 3 次正常经济出现率 0%。 必须 0%；兜底经济单独记录。
第 4-6 次分布 回到长期权重。 武器/模块/经济分布无明显偏离。

测试与验证建议

Contract Tests

  • 第 1 次标准 draft 不含模块和正常经济。
  • 前 3 次均含 weapon-progress 槽。
  • 第 4 次后无固定 weapon-progress 槽。
  • 任务奖励、特殊来源掉落、Bonus 路线不推进标准 draft count。
  • 标准战斗完成不会同时生成 Draft 和标准地面 weapon/module drop。

Persistence Tests

  • pending draft 选项序列化后恢复一致。
  • 恢复时失效选项转经济兜底，不重骰整轮。
  • 战斗回滚恢复 draft count 和 pending options。
  • 玩家选择奖励后清除 pending draft。

Simulation Report

  • 固定 seed。
  • 1000 次 run，前 6 次标准 draft。
  • HTML 报告写入 docs/reports/reward_draft_simulation_*.html。
  • 报告显示 PASS/WARN/FAIL。

Godot Gate

  • 优先 scene-backed focused tests。
  • 补充 shell --check-only。
  • 不要只用源码检查代替运行验证。

明确不做

  • 不调整敌人配置，除非后续奖励模拟或实战 smoke 证明早期输出/生存不足。
  • 不保留标准战斗完成后的自动地面 weapon/module drop；Draft 替代它。
  • 不删除特殊来源掉落，例如敌人、宝箱、事件、商店、任务或脚本指定奖励。
  • 不影响任务奖励、cell effect、task module 流程。
  • 第 4 次后不加 pity。
  • 不新增奖励面板阶段说明或模块首次出现提示。
  • 不做路线差异化奖励权重。
