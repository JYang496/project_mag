# 提示词 05：提取 Battle HUD 场景

请从当前大型 UI 场景中提取独立的 `UI/scenes/runtime/battle_hud.tscn` 和对应 `BattleHudView` 脚本，降低 `UI.gd` 对长节点路径的依赖，同时保持 `UI.gd` 作为业务层兼容门面。

## 开始前

检查主 UI 场景、`UI/scripts/UI.gd`、响应式布局、本地化代码，以及 WeaponSelector、HpLabel、Gold、Time、Resource、战斗状态标签、任务 HUD Host、提示 Host 的所有引用和信号。不要修改无关管理 UI。

## 实施要求

- 将计划指定的战斗 HUD 节点完整迁移到新场景，保留布局、主题、anchors、owner、脚本、唯一节点名和信号行为。
- 新建职责清晰的 `BattleHudView`，通过稳定字段或方法暴露外部需要的节点；避免业务调用方深入其内部节点树。
- `UI.gd` 改为持有 `battle_hud` 并访问其稳定接口，逐步消除对应的 `$GUI/...` 长硬路径。
- 不要求一次性迁移所有业务调用方；现有对 `UI.gd` 的调用应继续工作。
- Battle HUD 初始化完成时提供明确状态/信号，供 `world_ready` 使用。
- 延迟实例化或场景重入时不得产生重复信号、空引用或旧引用。
- 确保本地化和窗口尺寸变化能刷新新场景中的全部元素。

## 验收与验证

- 验证所有 HUD 数值、武器选择、任务、提示、战斗状态、光标相关表现。
- 验证不同窗口尺寸/宽高比下的响应式布局和至少两种语言切换。
- 搜索并报告已消除及仍保留的相关长节点路径。
- 运行 Godot `--check-only`，打开相关场景并进行聚焦运行验证。

完成后说明新视图 API、迁移节点、兼容层、剩余硬路径和验证结果。
