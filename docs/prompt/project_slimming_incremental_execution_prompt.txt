你在 Windows 本地 Godot 项目 `project_mag` 中工作。

任务：在不改变玩法、存档格式和玩家可见行为的前提下，分阶段执行项目瘦身。

本提示词是长期执行规范，不要求一次完成全部工作。每次运行只完成一个边界清晰、可独立验证、可独立回滚的批次。

## 开始前必须执行

```powershell
git status --short --branch
git rev-parse HEAD
```

如果工作区有其他人的改动：
- 不回滚、不覆盖、不格式化无关文件。
- 只修改当前批次需要的文件。
- 如果目标文件存在无法安全合并的并行改动，停止并报告冲突。

开始前阅读：
- `tests/README.md`
- `project.godot`
- 本提示词涉及的目标脚本及其直接测试
- `docs/plans/project_slimming_progress.md`，如果该文件存在

如果进度文件不存在，在第一个实施批次中创建它。进度文件只记录：
- 已完成批次
- 修改前后关键指标
- 执行过的测试及结果
- 已知风险和下一推荐批次

## 总体优先级

1. 降低代码复杂度
2. 缩短本地开发反馈时间
3. 运行内存专项延期

运行内存当前只保留两条约束：
- 不在热路径引入每帧 Dictionary、Array、Resource 或 RefCounted 临时分配。
- 不新增常驻 Autoload，除非有独立设计审查和可量化必要性。

不要创建内存压力测试、长期浸泡测试或专项内存优化分支。

## 不可违反的边界

- 保持玩法、数值、输入手感、UI 行为、存档兼容和资源路径兼容。
- 使用渐进式重构，不做一次性架构重写。
- 先补特征测试，再移动高风险逻辑。
- 不把 God Object 从主脚本转移到万能 Context、Dictionary 或服务定位器。
- 子系统不得读取或修改 Owner 的私有状态。
- 优先使用窄输入、明确配置和强类型结果。
- 引擎节点状态由对应 Node 应用；纯逻辑系统只计算结果或返回命令。
- 公开 API 首批保留薄门面，调用方迁移放到后续独立批次。
- 行数是软指标，职责边界、依赖方向和测试覆盖是主要验收标准。
- 不删除历史测试、报告、工具或资源，除非先完成引用与用途审计。
- 不以“代码已搬到新文件”作为完成标准；必须实际消除双向耦合或重复状态源。

## 每个批次的固定流程

1. 检查工作区和当前提交。
2. 阅读目标代码、直接调用方和现有测试。
3. 记录修改前指标：文件行数、函数数、状态字段数、反向 Owner 引用数，以及相关测试耗时。
4. 如果行为缺少保护，先增加最小特征测试。
5. 完成一个职责边界的修改。
6. 运行相关定向测试和 Godot 语法/资源检查。
7. 检查 `git diff --check` 和最终 `git diff`。
8. 更新进度文件。
9. 报告改动、指标变化、测试结果、剩余风险和下一推荐批次。
10. 当前批次失败时停止，不顺带进入下一批。

## 阶段一：UI.gd 收敛

目标：
- `UI/scripts/UI.gd` 最终只承担组件装配、生命周期协调和公开兼容门面。
- 软目标约 600 行。
- 不允许 Controller 反向读取或写入 `UI.gd` 私有字段。

### UI-1：任务模块对话框

新增专用 `TaskModuleDialogController`：
- 使用 `RefCounted`。
- 只注入现有 `ModalDialogController`。
- 不持有 `owner_ui`。
- 自己管理待确认回调、替换模块 ID、选中索引和动态按钮。

`UI.gd` 保留以下同名公开薄门面：
- `request_task_module_unassigned_confirmation`
- `request_task_module_replacement`

立即删除只服务于旧实现的六个 pending/custom-button 状态字段和内部辅助方法，不保留镜像状态。

测试门禁：
- 现有统一模态行为测试。
- 新 Controller 契约测试。
- 覆盖重复打开、确认、取消、右键、Esc、动态按钮清理和回调只执行一次。
- Godot 语法/资源检查。

### UI-2：管理菜单样式与布局

把主菜单样式、本地化文本刷新和布局职责收进现有管理 Helper/Controller。

规则：
- 重复静态样式使用共享 Theme/StyleBox 资源。
- 动态状态和运行时布局保留在 Controller。
- 不进行全量 `.tscn` 重建。
- 不改变当前视觉结果。

视觉门禁：
- 1280x720 英文
- 1280x720 简体中文
- 1920x1080 英文
- 1920x1080 简体中文
- 同时运行相关交互测试

### UI-3：统一模态协调

扩展现有 `ModalUiController`，使其成为唯一的模态注册、世界交互阻塞和取消协调者。

规则：
- 使用声明式有序注册表。
- 每个条目提供可见、阻塞和取消能力。
- 保持当前语义优先级，不根据 Z 顺序猜测。
- `UI.gd` 在启动时显式注册，在退出时清理。
- 删除 `UI.gd` 中重复的 fallback 和长取消链。
- 不新增 Modal Autoload。

测试必须锁定：
- 当前取消优先级。
- 模态阻塞规则。
- 右键与 Esc 行为。
- 注册、注销和失效节点处理。

### UI-4：HUD 刷新协调

用自包含刷新协调器替代当前对 `UI.gd` 私有 dirty 字段的回写。

协调器负责：
- 信号订阅与释放
- dirty 状态
- 刷新调度
- Presenter 调用
- 调试计数

调度规则：
- 生命、金币、库存、武器列表等离散状态由信号驱动。
- 热量、装填、冷却和计时等连续值最多按 10 Hz 采样。
- 对应 HUD 隐藏或状态无关时停止采样。
- 不产生每帧临时容器或对象。

### UI-5：输入协调

新增独立 `UiInputCoordinator`：
- 使用 `RefCounted`。
- 不反向修改 `UI.gd`。
- `UI.gd._input(event)` 只转发事件并根据返回值标记 consumed。
- 只使用传入的 `InputEvent`，不混用 `Input.is_action_just_pressed()`。
- 固定处理顺序：控件提示、模态取消、暂停、武器切换及相关 UI 动作。

保持以下动作语义不变：
- `ESC`
- `CANCEL`
- `SWITCH_LEFT`
- `SWITCH_RIGHT`
- `TOGGLE_CONTROLS`

### UI 阶段完成门禁

- 每个批次运行定向测试和语法检查。
- 阶段结束运行全量测试。
- 完成四组视觉截图检查。
- 比较 UI 刷新调试计数，不能出现无理由的刷新放大。
- `UI.gd` 接近 600 行软目标。
- Controller 不再回写 `UI.gd` 私有状态。

## 阶段二：Player.gd 收敛

目标：
- `Player/Mechas/scripts/Player.gd` 最终只承担引擎节点适配、组件装配、生命周期和公开兼容门面。
- 软目标约 700 行。
- 子系统不直接读取 Player 私有字段或调用 Player 私有方法。

### Player-1：移动系统试点

Player 继续持有：
- `movement_enabled`
- `moveto_enabled`
- `moveto_dest`
- CharacterBody2D 的位置与速度应用

移动系统只计算：
- 下一帧速度
- 是否到达目标
- 是否需要吸附位置
- 自动导航完成状态

数据传递：
- 使用一次分配、循环复用的强类型 `MovementFrameInput` 和 `MovementFrameResult`。
- 不使用每帧 Dictionary、Array 或新建对象。
- 系统不得持有 Player 引用。

测试门禁：
- 表驱动测试覆盖加速、减速、反向转向惩罚、PREPARE 禁止手动输入、自动导航到达和位置吸附。
- Dash 场景集成测试。
- 自动导航场景测试。
- 重复 Tick 的分配与性能检查。

### Player-2：主动技能运行时

保留现有公开门面，内部拆成：
- `SkillEnergyState`
- `PlayerSkillController`
- `WeaponActionController`

职责：
- `SkillEnergyState` 管理能量、上限、消耗和恢复。
- `PlayerSkillController` 管理默认技能实例与玩家技能施放。
- `WeaponActionController` 管理主武器主动技能、换弹判定和动作结果。

规则：
- 输入事件由 `UiInputCoordinator` 路由。
- `project.godot` 是技能输入动作的唯一配置源。
- 运行时只校验输入动作，不修改 `InputMap`。
- `WeaponActionController` 返回强类型 `WeaponActionResult`。
- Player 根据结果应用信号、Assist、被动事件和提示命令。
- 不直接访问 `GlobalVariables.ui`。

测试门禁：
- 重构前先补特征测试。
- 能量边界、恢复与消耗。
- 默认技能加载。
- 玩家技能施放。
- 主武器主动与换弹。
- 信号执行次数。
- Assist 后处理。
- 提示节流。
- InputMap 缺失校验。

### Player 后续顺序

完成前两个试点后，按以下顺序继续，每组独立批次：

1. 状态、拾取、相机系统
2. 视觉、武器轨道、碰撞体同步
3. 伤害反应与元素效果
4. 武器库存、武器被动、共享热量

每组遵循同一模式：
- 先记录当前行为。
- 先补特征测试。
- 移除 `_player` 私有状态访问。
- 使用窄输入和强类型结果。
- Player 负责应用 Node/引擎状态。
- 保留公开兼容门面。

不要直接改成 ECS，也不要创建万能 `PlayerContext`。

### Player 阶段完成门禁

- 每个批次运行对应战斗、武器和场景测试。
- 阶段结束运行全量测试及关键战斗 smoke。
- `Player.gd` 接近 700 行软目标。
- 子系统中不存在对 Player 私有字段/私有方法的直接访问。
- 输入手感、移动、Dash、伤害、武器切换、换弹和技能行为不变。

## 阶段三：活跃测试基础设施

建立声明式测试清单，包含：
- 测试 ID
- 测试入口类型：scene 或 script
- 入口路径
- 所属领域
- 依赖领域
- 是否可并行
- 是否写入 `user://`
- 超时时间

建立源码目录/文件到测试领域的保守映射。

执行策略：
- 先运行 Godot 语法/资源检查。
- 自动选择受影响测试。
- 使用有限并行的独立 Godot 进程。
- 每个 Worker 使用隔离的用户数据目录。
- 默认并行度保守设置，不得压垮本地机器。
- 保留现有单测试入口，不合并成单一巨型测试场景。

以下情况自动回退全量测试：
- `project.godot` 改动
- Autoload 或核心契约改动
- 测试清单自身改动
- 未映射的生产文件改动
- 依赖关系不确定

历史测试、报告和归档不在本阶段删除。先生成用途与引用审计，另立清理批次。

## 阶段四：本地启动与资源准备

优化目标顺序：
1. 定向测试耗时
2. 热启动
3. 冷导入

先建立基线：
- 使用隔离缓存测一次完整冷导入，不删除开发者当前 `.godot` 缓存。
- 热启动运行五次，记录中位数。
- 代表性定向测试运行五次，记录中位数。
- 记录加载资源数、错误数和阶段耗时。

启动策略：
- 启动菜单只加载设置和存档元数据。
- 进入世界前显式准备武器、机甲、经济、路线、Cell Effect、Task Module 等领域目录。
- 使用静态或生成的资源清单替代启动期目录扫描。
- 准备 API 必须显式、幂等并返回成功/失败状态。
- 世界切换必须等待必要依赖准备完成。
- 缺失资源必须明确报错，不以空数据静默继续。
- Getter 不得隐式触发昂贵加载。

不要为了启动速度新增常驻加载 Autoload。

## 阶段五：安全资源管线治理

只允许无视觉变化的处理：
- 审计重复字体与实际引用。
- 审计 `asset/images/characters/characters_move.zip` 是否属于运行时资产。
- 审计重复图片、未引用源文件和历史生成物。
- 统一同类 PNG、SVG、字体和音频的导入预设。
- 将只供制作使用的源文件移出运行时扫描路径或归档到明确目录。
- 删除前必须提供引用证据和恢复方式。

当前重点候选：
- `asset/fonts/NotoSansSC-Regular.ttf`
- `asset/fonts/NotoSansCJKsc-Regular.otf`
- `asset/images/characters/characters_move.zip`
- 大尺寸 Cell 图片

不要在本阶段：
- 降低图片分辨率
- 改变压缩造成可见失真
- 改字体覆盖范围
- 改音频质量
- 改资源路径而不提供兼容迁移

## 验收目标

代码复杂度：
- `UI.gd` 约 600 行软目标。
- `Player.gd` 约 700 行软目标。
- 新 Controller/System 职责单一。
- 不存在新的双向 Owner 私有状态依赖。

行为：
- 所有相关测试通过。
- 阶段里程碑全量测试通过。
- 存档与资源路径兼容。
- UI 四组视觉门禁通过。

本地反馈：
- 建立可重复基线后，定向测试、热启动和冷导入分别争取至少 20% 改善。
- 多次运行中位数不得出现超过 5% 的无解释回退。
- 如果 20% 目标无法达到，报告瓶颈证据，不通过扩大改动范围来伪造收益。

## 停止条件

出现以下任一情况时停止当前批次并报告：
- 需要改变玩法、数值、输入手感或存档格式。
- 现有测试无法确定预期行为。
- 目标文件存在无法安全合并的并行改动。
- 定向测试失败且无法证明与当前修改无关。
- 必须新增 Autoload、全局事件总线或万能 Context 才能继续。
- 资源删除缺少引用证据。
- 修改范围开始跨越两个以上独立职责。

7. 下一推荐批次
