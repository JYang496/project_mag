## 核心目标

把标准战斗完成后的主要奖励结算改成 Draft，而不是在地上自动生成 weapon/module drop。

Draft 的定义：标准战斗结束后，系统生成一组奖励选项，玩家从中选 1 个。它是标准战斗的主要结算奖励，不是地面掉落之外的额外奖励。

实现后不能出现：同一场标准战斗结束后，玩家既获得标准地面 weapon/module drop，又获得标准 Draft 三选一。

## 设计边界

本任务只影响“标准战斗完成结算奖励”。

需要替代：
- 当前 `RewardManager` 在 `BATTLE -> PREPARE` 后调用 `_spawn_completed_battle_drops()`，通过 `build_completed_battle_drop_rewards()` 自动生成标准战斗地面 weapon/module drop 的路径。
- 实现后，标准战斗完成应打开标准奖励 Draft，并停止这条自动地面掉落结算。

不要删除所有掉落系统。特殊来源掉落仍保留。

特殊来源指不来自 generic standard battle-complete settlement 的奖励来源，例如：
- 敌人自己指定的掉落
- 宝箱、事件、地图交互
- 商店购买
- 任务奖励、cell effect、task module
- Bonus 无战斗路线奖励
- 脚本明确放置或触发的特定奖励

这些特殊来源不被 Draft 替代，也不推进标准 Draft 次数。

## 锁定产品规则

1. 标准战斗 Draft 按“标准战斗奖励 Draft 打开次数”计数。
2. 任务奖励、Bonus 无战斗路线、特殊来源掉落、商店、仓库不推进这个计数。
3. Draft 次数在奖励选项生成并成功交给 UI 后递增。
4. 如果 UI 不可用，系统直接发第一个奖励，也算这次 Draft 已消费。
5. Draft 次数和 pending Draft 选项必须 run 内持久化，并参与战斗回滚/继续游戏恢复。
6. 新增独立 `RewardDraftRuntime` autoload 持有标准 Draft 状态。不要把状态塞进 `TaskRewardManager` 或 `RewardManager` 场景节点。
7. 前 3 次标准战斗 Draft 固定保留 1 个 `weapon-progress` 槽。
8. 前 3 次禁用正常经济权重。
9. 前 3 次允许最终防空池经济兜底，但只有武器/模块候选无法填满选项时才允许。
10. 第 1 次 Draft 模块概率 0%。
11. 第 2 次 Draft 模块概率 20%。
12. 第 3 次 Draft 使用正常长期模块概率。
13. 第 4 次及以后完全回到长期权重：武器约 50%，模块约 35%，经济约 15%。
14. 第 4 次后不保留固定 `weapon-progress` 槽。
15. 第 4 次后不加 pity。
16. 不调整敌人配置。只有后续奖励模拟或实战 smoke 证明早期压力错误时，才另行处理敌人。
17. 不新增奖励面板阶段说明，不显示 Early Draft 标签。
18. 不新增模块首次出现教学提示。

## Weapon-Progress 定义

`weapon-progress` 是统一候选池，不只是已有武器升级。

它包含：
- 已有武器升级候选：目标武器未满级。
- 已有武器融合候选：目标武器可 fuse。
- 新武器获得候选：普通武器 reward candidate，排除隐藏、零权重、已最终 fuse 的武器。

早期保底槽从统一 `weapon-progress` 池中按权重抽 1 个：
- 已有武器升级/融合使用目标武器定义的 rarity 和 drop_weight。
- 新武器获得使用 `WeaponDefinition` 的 rarity 和 drop_weight。

不要实现“优先已有武器升级”。已锁定为：已有武器进度和新武器获得同池竞争。

## RewardDraftRuntime 要求

新增 `autoload/RewardDraftRuntime.gd` 并注册 autoload。

职责：
- 保存 `standard_draft_count`
- 保存 pending standard Draft options 的序列化 payload
- 提供当前将要生成的是第几次标准 Draft
- 根据 `EconomyConfig` 返回阶段规则
- 在标准 Draft 交给 UI 前保存 pending options
- UI 打开成功或 fallback grant 后记录 Draft 已消费
- 玩家选择奖励成功后清除 pending Draft
- 提供 run 内状态保存/读取
- 提供 battle rollback snapshot 的 build/restore 接口
- 提供 reset runtime state

不要让 `RewardManager` 持有长期状态。

## Pending Draft 持久化

普通标准 Draft 一旦生成，选项要持久化，直到玩家选择或该次奖励明确结算。

恢复 pending Draft 时：
- 不重骰整轮。
- 如果某个选项无法重新绑定，只把该选项转换为经济兜底。
- 不要因为一个选项失效就重新生成整组奖励。

RewardInfo 序列化参考 `TaskRewardManager` 里已有逻辑：
- 保存 `kind`
- 保存 `item_id` / `item_level`
- 保存 `module_path` / `module_level`
- 保存 `gold` / `chip`
- 保存 `rarity`
- 保存 `reward_key`
- 保存 `target_weapon_id`
- 保存升级前后等级和展示名
- 不要直接 JSON 化 `PackedScene` 或 `WeakRef`
- 模块恢复时用 `module_path` 重新 load `PackedScene`
- weapon upgrade 恢复时用 `target_weapon_id` 绑定当前已装备武器
- 如果绑定失败，转换为经济兜底

## RewardManager 实现方向

当前标准战斗完成后自动地面掉落路径：
- `_on_phase_changed()`
- `_spawn_completed_battle_drops()`
- `build_completed_battle_drop_rewards()`

实现后：
- 标准战斗完成应请求/打开标准奖励 Draft。
- 不再调用 generic standard battle-complete ground drop 作为标准结算。
- 可以保留 drop 构建/生成能力给特殊来源使用，但必须让边界清楚，不能标准战斗完成时双发奖励。
- 如保留 `build_completed_battle_drop_rewards()`，请重命名、注释或 gate caller，明确它不再是标准战斗完成默认结算。

生成标准 Draft 时：
- 向 `RewardDraftRuntime` 获取 stage rules。
- early Draft 先抽/插入 1 个 weapon-progress 槽。
- 剩余槽位按当前阶段模块概率、经济可用性和候选池滚动。
- 保留 duplicate option key 去重。
- 前 3 次正常经济权重禁用。
- 武器/模块都无法填满时才允许 fallback economy。
- route `reward_option_count` 仍然生效：早期规则适用于所有标准战斗 Draft，不管 route 给 3 个还是更多选项；只保证 1 个 weapon-progress 槽。
