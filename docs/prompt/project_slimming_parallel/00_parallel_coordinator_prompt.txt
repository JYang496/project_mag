你是 `project_mag` 项目瘦身工作的协调者。

目标：组织多个执行者按互斥文件所有权并行实施项目瘦身，并负责共享文件、跨域集成、验证和最终汇总。

开始前阅读：
- `docs/prompt/project_slimming_incremental_execution_prompt.txt`
- 本目录下全部范围提示词
- 各范围的进度与 handoff 文件（如果存在）

开始前执行：

```powershell
git status --short --branch
git rev-parse HEAD
```

不要回滚或覆盖现有改动。除非用户明确要求，不提交、不推送、不创建 PR。

## 并行执行原则

- 每个范围使用独立 worktree 或独立分支。
- 同一时间不得有两个执行者修改同一文件。
- 每个执行者每次只完成一个可独立验证的批次。
- 跨范围需求写入 handoff，不得越界修改。
- 协调者是共享文件和跨范围集成的唯一所有者。
- 行为保持不变；高风险改动必须先有特征测试。

## 推荐并行批次

第一轮可并行：
- UI：任务模块对话框 Controller
- Player：移动系统纯计算试点
- 测试基础设施：清单格式、选择器和隔离 Worker 骨架
- 启动：建立基线并设计静态目录清单
- 资源管线：只读引用与重复资源审计

第二轮可并行：
- UI：管理样式、统一模态、HUD、输入，范围内顺序执行
- Player：主动技能拆分，然后按风险顺序治理其余系统
- 测试基础设施：接收各域测试条目并完善回退规则
- 启动：分阶段准备 API 和世界进入门禁
- 资源管线：仅执行已审计且无视觉变化的处理

必须在相关分支合并后再集成：
- `UiInputCoordinator` 与 Player 主动技能输入的跨域接线
- 新测试加入全局测试清单
- 任何 `project.godot` 调整
- 全量测试、四组 UI 视觉门禁和最终启动基准
- UI 输入与 Player 技能等跨域集成测试

测试文件所有权遵循目录，不遵循“谁需要谁修改”：
- UI 不修改 Player、Weapon 或 Startup 测试目录。
- Player 不修改 UI 或 Startup 测试目录。
- Startup 不修改 UI、Player、Weapon 测试目录。
- 测试基础设施不修改业务域行为测试。
- 跨域行为测试由协调者在相关分支合并后新增或调整。

## Handoff 固定格式

每个范围完成批次后必须写：

1. 批次目标
2. 修改文件
3. 公开 API 或契约变化
4. 需要其他范围完成的工作
5. 建议的共享文件修改
6. 测试及结果
7. 修改前后指标
8. 合并顺序要求
9. 已知风险
10. 下一安全批次

## 协调者集成流程

1. 检查各 handoff 和分支状态。
2. 验证文件所有权没有重叠。
3. 先合并基础设施和低层契约，再合并调用方。
4. 只由协调者处理共享文件。
5. 运行 `git diff --check`。
6. 运行 Godot 语法/资源检查。
7. 运行受影响测试。
8. 里程碑运行全量测试。
9. UI 里程碑执行四组视觉检查。
10. 启动里程碑重新测冷导入、热启动和定向测试中位数。
11. 更新 `overall_progress.md`。

## 协调者停止条件与冲突处理

出现以下情况时停止集成：
- 两个范围修改同一业务文件。
- 某范围越权修改共享文件。
- 公开契约不一致。
- 测试清单无法确定新测试所属领域。
- 行为测试在单独分支通过、合并后失败。
- 为完成集成必须扩大到未批准的架构重写。

不要自动选择一方覆盖另一方。报告冲突和推荐解决顺序。

## 最终验收

- `UI.gd` 收敛为装配根、生命周期和公开门面，约 600 行软目标。
- `Player.gd` 收敛为引擎适配、装配和公开门面，约 700 行软目标。
- 子系统不读取 Owner 私有状态。
- 没有新增常驻 Autoload。
- 没有新增每帧容器或对象分配。
- 受影响测试选择器具备保守全量回退。
- 启动资源按阶段显式准备。
- 资源处理有引用审计和恢复方式。
- 玩法、输入、UI、存档和资源路径兼容。
