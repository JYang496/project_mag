## 允许修改

- 上述明确列出的 Autoload 脚本
- 直接相关的数据资源清单
- `World/new_game_btn.gd`
- `World/continue.gd`
- 启动基准工具
- `tests/headless/startup/**`
- `tests/scenes/startup/**`
- `docs/plans/project_slimming/startup_progress.md`
- `docs/plans/project_slimming/handoffs/startup.md`

## 禁止修改

- `project.godot`
- `UI/**`
- `Player/**`
- 测试全局选择器与 `tests/README.md`
- UI、Player、Weapon 和跨域集成测试
- `asset/**`
- 其他范围进度文件

不得新增 Autoload。必须调整 `project.godot` 时写入 handoff。

## 工作顺序

### Startup-1：建立基线

- 使用隔离缓存测冷导入，不删除当前 `.godot`。
- 热启动运行五次。
- 代表性定向测试运行五次。
- 记录中位数、资源数、错误数和阶段耗时。

### Startup-2：静态资源目录

- 为武器、机甲、经济、路线、Cell Effect 和 Task Module 建立明确清单。
- 开发环境可提供清单一致性验证。
- 运行时启动路径不重复扫描目录。
- 缺失、重复 ID 和无效类型必须明确失败。

### Startup-3：分阶段准备

菜单阶段只准备：
- 设置
- 本地化基础
- 存档元数据

进入世界前显式准备：
- 武器
- 机甲
- 经济
- 路线
- Cell Effect
- Task Module
- 其他世界运行必需目录

准备 API：
- 显式
- 幂等
- 返回状态和错误
- 声明依赖顺序
- Getter 不隐式触发昂贵加载

世界切换必须等待必要准备完成。失败时阻止进入并报告，不以空数据继续。

### Startup-4：重新测量

- 重复相同基线。
- 目标热启动和冷导入各至少改善 20%。
- 多次中位数不得无解释回退超过 5%。
- 如果收益不足，报告证据，不扩大到 UI、Player 或资源有损修改。

## 并行协作

- 不修改测试全局清单；提供建议条目到 handoff。
- 不修改资源源文件；把可疑资源路径交给资源管线范围。
- 不修改 `project.godot`；把必要设置交给协调者。

## 停止条件

- 分阶段加载改变存档恢复结果。
- 世界可能在资源未准备时继续。
- 必须新增 Autoload。
- 静态清单与运行资源无法一致验证。
- 优化需要修改 UI 或 Player。
