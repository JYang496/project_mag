结论摘要

本次遍历覆盖项目内 GDScript、场景/资源文件、资产导入配对和现有测试工具。项目已经具备较好的模块化雏形，例如 EnemyRegistry、ObjectPool、HUD dirty refresh、多个 headless 测试脚本，但仍有若干高收益优化节点集中在数据索引、运行时读取/保存、对象生命周期、单体脚本拆分和资产管理上。

最高优先级建议：继续推进 EnemyRegistry 替代战斗热路径上的 get_nodes_in_group()，补齐对象池真实复用能力，降低 Player.gd、UI.gd、enemy_spawner.gd 等大文件的职责密度。

重点优化节点

优先级 节点 证据位置 影响 建议
高 敌人/收集物全局查询仍散落在运行时路径 Utility/enemy_spawner.gd、InventoryData.gd、wmod_runtime_utils.gd、area_effect.gd 战斗中敌人数量上升后，全量 group 扫描会随对象数线性增长。项目已有 EnemyRegistry 和 benchmark，说明该问题已经被局部识别。 把攻击、范围效果、任务目标、剩余金币统计的敌人查询统一改为 EnemyRegistry。收集物可增加 CollectableRegistry 或在 EnemySpawner 中维护金币计数，避免战斗结束时重复扫 collectables。
高 对象池存在但当前未复用 autoload/ObjectPool.gd acquire() 每次仍 scene.instantiate()，release() 直接 queue_free()。弹体、掉落、伤害数字等高频对象无法获得池化收益。 先选择 projectile、coin/chip/drop、hit label 三类对象接入真实缓存；定义 _on_acquired_from_pool() 和 _on_before_pooled() 生命周期清理协议；限制单场景缓存数量并加统计输出。
高 核心脚本职责过重 Player.gd 约 82KB、UI.gd 约 74KB、enemy_spawner.gd 约 50KB、rest_area.gd 约 46KB 这些文件同时处理输入、状态、表现、数据恢复、调试和 UI 协调，变更成本高，测试粒度难以收窄。 优先按已有模式拆分：Player 的元素状态/被动/自动拾取/武器轨道继续下沉到 system；UI 的弹窗加载、管理界面入口、光标表现进一步移到 controller；EnemySpawner 拆出预算、生成点选择、金币预算三块。
中 资源加载和目录扫描逻辑重复 DataHandler.gd、CellEffectRuntime.gd、CellTaskModuleRuntime.gd、RunRouteManager.gd 多个 autoload 都手写 DirAccess.open() + load() + fallback 路径。行为不一致时容易出现导出包遗漏、重复注册或排序不稳定。 抽出 ResourceCatalog 辅助：输入目录、后缀、fallback 列表、id getter，输出稳定排序后的资源字典和 warning 列表。DataHandler 可额外维护 scene_path 到 weapon_id 的反向索引。
中 运行时状态保存过于即时 InventoryData.gd 的 store/equip/merge/transaction 多处调用 save_runtime_state() 连续装备、升级、交易时可能多次写 user://equipment_runtime_state.json。虽然数据量不大，但会增加 I/O 噪声，也让事务边界不清晰。 引入脏标记和延迟保存，例如 mark_runtime_state_dirty() + 0.2 秒 debounce；关键节点和关闭通知仍强制 flush。保存失败时返回错误或写 warning。
中 深拷贝和数组复制使用偏多 匹配到 duplicate(true) 50 处、duplicate() 38 处 深拷贝在状态快照、交易、效果参数中是安全的，但在热路径或 UI 刷新中容易产生不必要分配。 区分只读快照和可变拷贝。对只读配置资源返回 typed arrays 或 Resource 引用；对外暴露需要保护的集合时保留浅拷贝，深拷贝仅用于持久化/事务隔离。
中 数据结构可增加反向索引 DataHandler.get_weapon_id_from_scene_path() 遍历 GlobalVariables.weapon_list.keys() 武器存取、奖励结算、仓库逻辑经常根据实例/scene_path 找 weapon_id，每次遍历会放大调用成本，也让隐藏资源处理分散。 加载武器数据时同时维护 weapon_id_by_scene_path。模块也可维护 module_by_scene_path，用于重复模块检测和恢复。
低 调试输出默认值偏激进 Utility/enemy_spawner.gd 中 debug_print_spawn_stats、debug_print_kill_gold_stats 默认 true 开发期有帮助，但正式运行或自动测试中容易污染输出，影响定位真正错误。 按 build/profile 控制默认开关，或统一接入 debug flag/autoload logger。测试脚本可显式打开需要的输出。
低 资产导入存在孤立文件 asset/images/weapons/Remove_background_to_create_transparent_PNG-1777246945516.png.import 源 PNG 缺失但 import 文件仍在，可能来自删除资产后的残留。长期会让资源目录和导出结果难以审计。 确认该资产不再被引用后删除孤立 .import。增加一个轻量资源检查脚本，CI 中扫描孤立 .uid、.import、缺失 preload/load 目标。

数据与读取方式

资源目录加载

DataHandler、RunRouteManager、CellTaskModuleRuntime、CellEffectRuntime 都采用目录扫描加载资源的模式。该方式适合内容扩展，但建议统一排序、统一 fallback、统一 warning 格式，以减少导出环境下的非确定性。

持久化状态

InventoryData 和 TaskRewardManager 使用 JSON 保存运行时状态。当前保存函数较直接，建议加入 schema version、保存失败 warning、延迟写入和原子替换策略。对装备仓库这种关键数据，先写临时文件再替换正式文件更稳妥。

索引结构

建议新增以下索引结构：

  • weapon_id_by_scene_path: Dictionary[String, String]，替代按 weapon_list 全量遍历。
  • module_definition_by_scene_path 或模块 scene 缓存，服务重复检测、商店、奖励恢复。
  • collectables_by_type 或轻量 CollectableRegistry，替代战斗结算时按 group 扫描金币。
  • 敌人空间查询若半径查询增多，可在 EnemyRegistry 外增加网格索引；项目已有 benchmark 对 grid 方案做了雏形验证。

建议实施顺序

  1. 先做低风险索引优化：在 DataHandler 构建 scene_path 反向索引，并用测试覆盖武器恢复、奖励结算、仓库换装。
  2. 把战斗热路径的敌人查询统一切到 EnemyRegistry，复用现有 run_enemy_registry_many_enemy_benchmark_headless.gd 验证。
  3. 让 ObjectPool 真正缓存 projectile、coin/chip、hit label 三类对象，并增加泄漏/重复 release 检查。
  4. 给 InventoryData.save_runtime_state() 增加脏标记和 debounce，避免连续操作反复写 JSON。
  5. 逐步拆分大文件。每次只迁移一个职责，并保持旧公开方法兼容，降低 UI 和测试联动风险。
  6. 补充资源健康检查脚本，扫描孤立 import、缺失 load/preload 目标、重复资源 id、空 id。
