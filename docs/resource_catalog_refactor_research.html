资源加载与目录扫描优化研究

生成日期：2026-06-25；范围：DataHandler、CellEffectRuntime、CellTaskModuleRuntime、RunRouteManager

结论

四处 autoload 的重复点主要集中在 DirAccess.open()、后缀过滤、排序、fallback 路径处理。本次新增 ResourceCatalog，统一输出稳定排序后的资源路径；业务侧仍保留各自的类型校验、id 校验和注册逻辑。

本次改造

  • 新增 autoload/ResourceCatalog.gd，提供 collect_resource_paths(directory, extension, fallback_paths, warn_missing_directory, context)。
  • DataHandler.gd 的 weapon、mecha、weapon branch、weapon passive 加载改为统一路径收集。
  • DataHandler.gd 新增 _weapon_id_by_scene_path，加载武器资源时建立 scene_path 到 weapon_id 的反向索引。
  • CellEffectRuntime.gd 和 CellTaskModuleRuntime.gd 改为通过 ResourceCatalog 稳定列出定义资源。
  • RunRouteManager.gd 移除手写排序目录扫描，改用同一工具，并保留原有 fallback route 行为。
  • 新增 World/Test/run_resource_catalog_test_headless.gd，验证路径排序、定义加载和武器反向索引。

设计边界

问题 处理方式 原因
是否让 ResourceCatalog 直接按 id 注册资源 未做 各业务资源的 id 字段、类型、分组结构不同。路径收集可通用，业务校验保持在原 autoload 内更清晰。
是否删除 fallback 保留 fallback 对导出包和历史资源路径有保护作用，统一工具只改变路径来源，不改变容错策略。
是否依赖 class_name ResourceCatalog 调用方使用显式 preload() 这些脚本都是早期 autoload，显式 preload 避免全局类注册顺序问题。

验证结果

  • Godot --headless --path . --quit：通过。
  • World/Test/run_resource_catalog_test_headless.gd：通过。
  • World/Test/run_weapon_delivery_type_test_headless.gd：通过，验证 15 个武器分类与模块。该既有测试退出时仍有资源泄漏警告，属于原测试清理问题。
