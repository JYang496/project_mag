# MagArena 像素风渲染一致性迁移计划

## 1. 文档目的

本文档记录 `MagArena` 从当前混合型高清科幻美术逐步迁移到现代像素风时的推荐落地顺序。

迁移目标不是一次性重做全部素材，而是在不改变核心玩法、碰撞、数据结构和动画状态机的前提下，先建立稳定的全局渲染规则，再按批次替换美术资源。

推荐的总体方向是：

- 战斗世界采用现代高清像素风；
- HUD、菜单和中文正文保持较高分辨率，以保证可读性；
- 保留现有蓝黑、白色装甲、青色能量、红色危险和金色奖励的配色语义；
- 优先统一像素密度、过滤、缩放、摄像机、程序化绘制和特效，再批量生产素材。

## 2. 核心原则

1. 先建立规范，后制作大批量素材。
2. 游戏逻辑继续使用浮点坐标，像素吸附只作用于视觉表现。
3. 战斗世界与 UI 分层处理，不强迫中文信息界面运行在极低分辨率下。
4. 新旧素材允许短期共存，但必须设定过渡期限和统一的临时处理方式。
5. 每个阶段均应形成可运行、可比较、可回退的版本。
6. 不在缺少视觉切片验证的情况下直接重制全部角色动画或模块图标。

## 3. 建议的目标渲染结构

首选方案：

```text
Main
├─ SubViewportContainer
│  └─ BattleViewport（建议从 640×360 开始验证）
│     └─ World
└─ CanvasLayer
   └─ UI（维持 1280×720 设计基准）
```

建议以 640×360 作为战斗世界的第一套验证规格，再以 Nearest 方式放大到 1280×720。UI 继续使用 1280×720 基准，中文正文保留平滑字体。

最终是否采用 SubViewport，应在垂直切片验证后确定。若现有鼠标坐标、世界交互或性能成本不适合该结构，可以退回到原生 1280×720 渲染，并对世界视觉节点实施统一像素吸附。

## 4. 分阶段落地顺序

### 阶段 0：建立基线

目的：保存迁移前可比较的视觉和运行基线。

任务：

- 截取主菜单、普通战斗、高密度战斗、休整区、商店和构筑界面的基准画面；
- 记录玩家、普通敌人、精英敌人、弹体和地块当前屏幕占用尺寸；
- 记录 1280×720、1920×1080、2560×1440 和至少一种 16:10 窗口下的表现；
- 记录慢速摄像机移动、高速玩家移动、击退和大量粒子时的表现；
- 建立一份代表性资源清单，避免测试时随意更换对象。

完成标准：

- 所有后续阶段都能与同一批场景和截图直接比较；
- 当前输入、瞄准、碰撞和 UI 布局行为已有记录。

### 阶段 1：确定像素美术规范

目的：在生产素材前锁定可执行的视觉约束。

需要决定：

- 战斗世界内部渲染分辨率；
- 一个美术像素对应的世界尺寸；
- 玩家、普通敌人、精英敌人、弹体、地块和图标的目标画布规格；
- 普通单位和大型单位的轮廓宽度；
- 调色板及各颜色的玩法语义；
- 阴影、亮面、高光和能量核心的层级数量；
- 世界素材与 UI 素材分别使用 Nearest 还是 Linear；
- 中文正文、数字、短标题是否分别使用平滑字体和像素字体；
- 发光、透明度、抖色和动画帧率的允许范围。

建议的第一版规格：

| 内容 | 建议规格 |
|---|---|
| 战斗世界 | 640×360 内部画面 |
| UI | 1280×720 设计基准 |
| 玩家 | 高 80–112 像素 |
| 普通敌人 | 48–72 像素 |
| 精英敌人 | 80–112 像素 |
| 武器图标 | 32×32 或 48×48 |
| 模块图标 | 24×24 或 32×32 |
| 地块纹理单元 | 32、48 或 64 像素 |

完成标准：

- 形成一页可供程序和美术共同执行的风格规范；
- 所有常用单位都有目标尺寸；
- 不再以“看起来差不多”为理由随意决定像素密度。

### 阶段 2：建立垂直切片

目的：用最小但完整的战斗组合验证视觉方向。

切片至少包含：

- 1 个玩家静态形象或少量关键移动帧；
- 3 个普通敌人；
- 1 个精英敌人；
- 2–3 把武器及其弹体；
- 默认金属地块；
- 1 个拾取物；
- HP、能量和武器槽 HUD；
- 爆炸、命中、光束各 1 个；
- 1 个地块任务或危险区域标记。

本阶段不应重制全部 42 帧玩家移动动画，也不应一次性转换全部模块 SVG。

完成标准：

- 切片中的玩家、敌人、地块、弹体、特效和 HUD 看起来属于同一套视觉系统；
- 在正常战斗密度下仍能一眼区分玩家、敌人、危险区和奖励；
- 确认现代高清像素风适合当前玩法后，才进入渲染架构调整。

### 阶段 3：统一纹理导入与过滤

目的：消除模糊、边缘插值和不同节点之间的过滤差异。

任务：

- 为像素纹理统一使用 Nearest；
- 对像素纹理关闭 mipmap 和不必要的各向异性过滤；
- 避免有损压缩破坏小尺寸轮廓；
- 将过滤设置尽量放在共同的世界视觉根节点继承；
- 逐步清理 Sprite2D、TextureRect 和 AnimatedSprite2D 上的零散覆盖；
- 移除玩家代码中对 `TEXTURE_FILTER_LINEAR_WITH_MIPMAPS` 的强制设置；
- 将字体、过渡期高清 UI 和像素素材分为不同导入策略。

重点检查：

- `Player/Mechas/scripts/Player.gd`；
- `Player/Mechas/scenes/Player.tscn`；
- `Board/Cells/cell.tscn`；
- `Objects/loots/coin.tscn`；
- `Objects/loots/loot_box.tscn`；
- `Player/Weapons/Projectiles/*.tscn`。

完成标准：

- 像素素材静止和缩放时保持清晰；
- 不再依赖每个素材作者手工修正节点过滤；
- 字体不会因全局 Nearest 设置而失去可读性。

### 阶段 4：统一摄像机、坐标和窗口缩放

目的：消除亚像素抖动、不均匀像素和不同分辨率下的拉伸。

任务：

- 验证 640×360 BattleViewport 方案；
- 将战斗世界以等比 Nearest 方式放大；
- 禁止 X、Y 使用不同倍率拉伸；
- 保留物理与移动逻辑的浮点坐标；
- 仅对 VisualRoot、最终摄像机或低分辨率输出实施像素量化；
- 规定精灵允许的整数缩放档位；
- 明确非 16:9 屏幕使用扩展视野、主题边框还是裁切；
- 验证摄像机跟随、震屏、击退和高速运动。

如果采用 SubViewport，还必须验证：

- 窗口鼠标坐标到战斗世界坐标的转换；
- 鼠标攻击和武器朝向；
- 战斗光标；
- 地块点击与任务区域交互；
- 世界空间标记与屏幕空间 HUD 的边界；
- 截图、暂停和全屏后处理。

完成标准：

- 慢速移动时没有明显的像素列跳变；
- 地块连接处没有闪缝；
- 武器、枪口和角色主体没有周期性错位；
- 鼠标瞄准和点击位置与迁移前一致；
- 常用分辨率下像素保持正方形。

### 阶段 5：统一程序化绘制

目的：让 `_draw()`、Line2D 和运行时生成图形遵循像素语言。

任务：

- 将非整数线宽逐步改为有规则的整数档位；
- 将关键坐标、半径和边界吸附到绘制网格；
- 世界内图形尽量禁用抗锯齿；
- 减少连续透明度变化，改用有限透明度档位；
- 将重要的圆环、箭头和徽章改成预制像素纹理；
- 对非关键动态图形保留代码绘制，但限制分段数和线宽；
- 区分世界内像素图形与可保持高清的 UI 文字。

重点检查：

- `Board/Cells/cell_activation_visual.gd`；
- `Board/Cells/cell.gd`；
- `UI/scripts/components/combat_resource_meter.gd`；
- `UI/scripts/components/player_health_meter.gd`；
- `UI/scripts/battle_contract_card.gd`；
- `World/start_battle_button.gd`。

完成标准：

- 地块边界、任务标记、攻击预警和 HUD 图标不再呈现另一套高清矢量语言；
- 圆形图形不是简单的低边数多边形，而是符合像素规范的图案；
- 同一层级的边框使用一致的线宽。

### 阶段 6：统一 Shader、光束和粒子

目的：让连续渲染效果与像素主体协调。

任务：

- 对需要的 Shader 加入 UV 量化；
- 对扫描、流动和噪声动画加入时间量化；
- 根据效果类型使用有限颜色或亮度档位；
- 缩短过宽的平滑边缘，避免主体全部由柔光构成；
- 将光束拆分为清晰像素核心、主题色主体和受控柔光外层；
- 粒子尺寸限制为少量档位；
- 粒子旋转可量化为 45° 或 90° 档位；
- 爆炸、火焰、冰霜等逐帧效果优先使用人工修整的像素动画。

重点检查：

- `Shaders/rest_area_zone_ground.gdshader`；
- `Shaders/layered_area_ground.gdshader`；
- `Shaders/ground_connected_effect.gdshader`；
- `Shaders/ground_cone_flow.gdshader`；
- `Shaders/ground_beam_flow.gdshader`；
- `Player/Weapons/Projectiles/beam.tscn`；
- `Player/Weapons/Projectiles/beam_blast.tscn`。

完成标准：

- 光束、地面区域和粒子不会显得比角色精细一个时代；
- 发光效果仍然醒目，但核心轮廓清晰；
- Shader 动画节奏符合所选像素动画帧率。

### 阶段 7：按优先级批量替换素材

推荐顺序：

1. 敌人、弹体和拾取物；
2. 地块与战场环境；
3. 武器与战斗特效；
4. 玩家角色及完整移动动画；
5. 战斗 HUD；
6. 主菜单、休整区、商店、仓库等外围 UI；
7. 55 个模块 SVG 及其余构筑图标。

每一批素材必须：

- 遵循既定画布尺寸；
- 使用项目调色板；
- 使用规定轮廓宽度；
- 保持现有资源锚点、枪口位置和碰撞语义；
- 在代表性战斗场景中验证，而不是只查看单张原图；
- 完成后再进入下一批。

完成标准：

- 高频战斗画面优先达到统一风格；
- 玩家完整动画和大量图标只在规范稳定后投入制作；
- 新素材替换不要求修改核心玩法逻辑。

### 阶段 8：清理过渡资产与兼容逻辑

目的：结束长期的新旧风格混用。

任务：

- 删除不再使用的高清资源引用；
- 清除临时限色、降采样或统一色调 Shader；
- 清理节点级过滤和缩放兼容代码；
- 检查遗漏的占位图、备份图和缺失图标；
- 确认所有资源使用稳定、统一的命名；
- 更新美术资源制作与导入说明。

完成标准：

- 运行场景中不存在依赖临时过渡效果才能协调的素材；
- 不再同时维护两套无必要的正式资源；
- 新增素材有明确的制作、导入和验收流程。

## 5. 每批素材的验收清单

- [ ] 使用规定的画布尺寸和像素密度；
- [ ] 使用项目调色板或经过批准的扩展色；
- [ ] 轮廓宽度符合对象层级；
- [ ] 场景中采用整数倍率显示；
- [ ] 未意外启用 Linear、mipmap 或有损压缩；
- [ ] 慢速移动时没有明显抖动或像素列闪烁；
- [ ] 地块和九宫格边缘没有接缝；
- [ ] 枪口、弹体出生点、阴影和碰撞位置正确；
- [ ] 玩家、敌人、危险区和奖励保持清晰区分；
- [ ] 光束与粒子的精细度不压过主体；
- [ ] 1280×720、1080p、1440p 和非 16:9 窗口表现可接受；
- [ ] 中文正文仍然清晰；
- [ ] 高密度战斗下仍有足够的信息层级；
- [ ] 相关自动化测试和项目编译检查通过；
- [ ] 保存了迁移前后的对比截图。

## 6. 测试与验证建议

每次涉及渲染架构、场景结构、输入坐标或 Shader 的修改后，至少执行：

```powershell
& 'E:\Godot_v4.7.1-stable_win64\Godot_v4.7.1-stable_win64_console.exe' --headless --path . --check-only --quit
```

如果修改影响已有运行领域，应根据 `tests/README.md` 使用受影响测试选择器和现有测试基础设施。纯素材替换也应至少打开代表性战斗场景进行视觉验证。

视觉验证优先覆盖：

- 静止画面；
- 慢速运动；
- 高速运动和击退；
- 摄像机移动与震屏；
- 大量敌人和粒子；
- 窗口尺寸变化；
- 鼠标瞄准和地块交互；
- 暂停、菜单和战斗 HUD 叠加。

## 7. 回滚与风险控制

- 每个阶段单独提交，避免渲染架构与大批素材混在同一个变更中；
- 新素材优先沿用旧资源路径或通过集中映射替换，减少场景逻辑变动；
- 保留垂直切片作为视觉基准，不因单张素材效果好就改变全局规范；
- 若 SubViewport 导致输入、性能或兼容问题，应能退回原生分辨率加视觉节点吸附方案；
- 过渡 Shader 只用于短期统一旧素材，不能成为永久依赖；
- 任何会改变碰撞、攻击范围或命中反馈的视觉调整，都应单独验证玩法一致性。

## 8. 关键决策记录

以下项目在进入大规模素材生产前必须明确，并在本文档中更新：

| 决策项 | 当前建议 | 最终决定 |
|---|---|---|
| 战斗内部渲染分辨率 | 640×360 | 待垂直切片验证 |
| UI 设计分辨率 | 1280×720 | 待确认 |
| 战斗世界渲染方式 | SubViewport + Nearest 放大 | 待技术验证 |
| UI 风格 | 像素面板与图标 + 平滑中文正文 | 待确认 |
| 玩家目标高度 | 80–112 像素 | 待切片确认 |
| 普通敌人尺寸 | 48–72 像素 | 待切片确认 |
| 像素素材过滤 | Nearest，无 mipmap | 待实施 |
| 非 16:9 策略 | 等比缩放并扩展视野或加边框 | 待验证 |
| 过渡旧素材策略 | 临时限色或低分辨率渲染 | 按需使用 |

## 9. 下一步

下一次正式开始迁移时，应从“阶段 0：建立基线”和“阶段 1：确定像素美术规范”开始，而不是直接批量生成或缩小现有素材。

阶段 1 完成后，先制作阶段 2 的垂直切片；只有切片通过画面辨识度、运动稳定性、输入准确性和多分辨率验证，才进入全局渲染架构调整及批量替换。
