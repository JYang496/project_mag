Prompt 1：建立武器被动统一状态接口

在当前 Godot 项目中，为武器被动 UI 做准备。不要重构现有触发逻辑，只在 Player/Weapons/Core/weapon.gd 增加统一读取接口。

目标：

新增 get_passive_status() -> Dictionary
新增兼容 API：
is_passive_ready()
notify_passive_triggered(cooldown_sec := 0.0)
refresh_passive_on_reload()
保留现有 is_offhand_skill_ready()、notify_offhand_skill_triggered() 行为，避免破坏现有武器逻辑。
默认 get_passive_status() 返回空/不可用状态，例如：
id
display_name
state
progress
current
required
ready
trigger_hint
refresh_hint
验收：

现有武器被动仍能触发。
现有 reload 刷新副手触发的逻辑不变。
UI 层可以对任意武器调用 get_passive_status() 而不报错。

Prompt 2：把 offhand_skill_ready 语义包成武器被动

将当前 _offhand_skill_ready 的语义包装为“武器被动是否可触发”。

目标：

内部变量可暂时不改名，但外部新代码应使用 passive 命名 API。
notify_offhand_skill_triggered() 内部调用或等价于 notify_passive_triggered()。
_refresh_offhand_skill_on_reload() 内部调用或等价于 refresh_passive_on_reload()。
所有现有武器脚本可以暂时继续调用旧 API。
验收：

不做大规模命名替换。
不改变任何武器数值和触发时机。
之后 UI 和新功能只依赖 passive API。

Prompt 3：标准化被动触发事件 detail

统一 emit_passive_trigger(...) 的 detail 字段，方便 UI、日志、调试读取。

目标：

在 emit_passive_trigger() 中补齐标准字段，而不是要求每把武器手写完整字段。
建议字段：
passive_id
trigger_type
refresh_type
state_after_trigger
passive_scope
如果调用方已经传了对应字段，不覆盖。
保留现有字段，例如 trigger、refresh，兼容旧逻辑。
验收：

现有 debug print 信息更完整。
旧的 detail.get("trigger")、detail.get("refresh") 仍可用。
UI 可统一读取 passive_id 和 refresh_type。

Prompt 4：为喷火器实现被动状态读取

为 Player/Weapons/flamethrower.gd 实现 get_passive_status()，用于显示“热量准备”进度。

当前逻辑：

主武器每次射击累计 _heat_prepared_accumulated_heat
达到 heat_max_value
开始换弹时立刻触发 flamethrower_heat_prepared
目标：

暴露：
id = "flamethrower_heat_prepared"
state = "charging" / "ready_pending_action" / "waiting_refresh" / "inactive"
progress = accumulated / required
current = _heat_prepared_accumulated_heat
required = heat_max_value
trigger_hint = "reload_started"
refresh_hint = "reload_finished"
条件已满足但还没换弹时，状态必须是 ready_pending_action。
验收：

UI 可以知道喷火器已经满足触发条件。
喷火器实际触发时机是开始换弹。
换弹完成后状态能正确恢复下一轮累计。

Prompt 5：为计数型被动实现状态读取

为以下武器实现 get_passive_status()：

Pistol：连续移动 6s
Spear Launcher：同一投射物命中 3 个不同目标
Charged Blaster：同一次光束命中 3 个不同目标
Plasma Lance：成功耗热攻击 3 次，换弹完成触发
目标：

每个武器返回统一结构。
计数型状态使用：
state = "charging" 或 "ready_pending_action"
current
required
progress
如果被动已经触发且等待换弹刷新，返回 waiting_refresh。
如果武器不是主武器，返回 inactive。
验收：

UI 能显示 2/3、4.2/6s 这类进度。
不改变任何触发条件。
不把一次性局部状态错误地暴露成长期状态。

Prompt 6：为阈值型被动实现状态读取

为以下武器实现轻量状态读取：

Sniper：远距离命中 >= 400
Shotgun：近距离命中 < 180
Dash Blade：冲刺距离 >= 攻击范围 75%
目标：

不强行做进度条。
返回 UI 用于提示条件：
state = "ready" / "waiting_refresh" / "inactive"
condition_type = "distance_threshold"
required
comparison = ">=" 或 "<"
如果能读取当前瞄准/目标距离，可提供 current 和 progress；不能稳定读取就不要伪造。
验收：

UI 至少知道这些被动是否可触发。
UI 可以在描述层显示触发条件。
不引入昂贵的目标扫描。

Prompt 7：为事件反应型被动实现状态读取

为以下武器实现轻量状态读取：

Orbit：玩家受伤触发
Chainsaw Launcher：投射物撞墙触发
Rocket Launcher：击杀后附近有其他敌人触发
Glacier Projector：冰冻命中触发，带 ICD
目标：

返回 state = "ready" / "cooldown" / "waiting_refresh" / "inactive"。
Glacier Projector 要暴露 ICD 剩余时间或进度。
事件反应型不需要常驻进度条，只需要 ready/cooldown 状态。
验收：

UI 能显示“可用/不可用”。
冰川投射器能显示短冷却。
不改变任何事件监听逻辑。

Prompt 8：接入全装备武器被动 UI 读取层

在 UI 层增加一个只读 presenter，用来读取所有已装备武器的 get_passive_status()。

目标：

遍历 PlayerData.player_weapon_list 中所有武器。
对每把武器调用 get_passive_status()。
UI 不直接判断具体武器类型。
UI 只消费统一字段：
weapon_id / weapon_name
is_main_weapon
state
progress
current
required
ready
trigger_hint
refresh_hint
主武器被动正常显示进度和可触发状态。
非主武器被动也显示，但状态应明确为：
inactive
inactive_reason = "not_main_weapon"
非主武器可以保留触发条件摘要，但不显示成“马上可触发”。
验收：

UI 同时列出所有已装备武器的被动。
当前主武器高亮。
非主武器灰显或低强调显示。
切换主武器后，各武器被动状态正确切换。
喷火器作为主武器时能显示累计热量进度；作为非主武器时显示休眠。
被动触发后对应武器进入 waiting_refresh。
换弹刷新后对应武器恢复可用/充能状态。
